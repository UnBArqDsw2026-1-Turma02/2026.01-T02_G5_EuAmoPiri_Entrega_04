# Módulo 07 — Comentários e Reações em Relatos

Documento de reutilização de software para **RF12** (comentários) e **RF13** (reações emoji) no backend Eu Amo Piri. A equipe **não adicionou bibliotecas npm novas** para este escopo; a reutilização concentra-se na **stack existente** (Prisma, Passport JWT, Express, Vitest) e em **código interno** já produzido para relatos e autenticação.

---

## 1. O que foi implementado

| RF | Funcionalidade | Endpoint |
|----|----------------|----------|
| RF12 | Listar comentários de um relato | `GET /places/:placeId/experiences/:experienceId/comments` |
| RF12 | Publicar comentário (turista) | `POST .../comments` |
| RF13 | Toggle de reação heart/like | `POST .../react` |
| — | Agregação na listagem de relatos | `GET /places/:placeId/experiences` — `commentsCount`, `reactions`, `myReaction` |

Models Prisma: `ExperienceComment`, `ExperienceReaction`, enum `ReactionType` (`HEART`, `LIKE`).

---

## 2. Por que foi implementado

Os relatos de experiência são o núcleo colaborativo do Eu Amo Piri. Comentários e reações aumentam engajamento sem exigir novo relato completo. A equipe manteve a **hierarquia Local → Relato → Comentário/Reação** e reutilizou a arquitetura MVC já validada em `experience*` para reduzir risco e tempo de entrega.

---

## 3. Reutilização de software

> RF12/RF13 ilustram **reutilização em cascata**: stack já instalada + padrão MVC + utilitários internos — a equipe entregou comentários e reações sem novas dependências npm.

### 3.1 Prisma ORM

| Aspecto | Detalhe |
|---------|---------|
| **O que faz** | Persiste comentários e reações; FKs com `ON DELETE CASCADE`; `groupBy` para contadores. |
| **Por que a equipe reutilizou** | Mesma stack de todos os módulos — tipagem gerada, migrations versionadas. |
| **Facilidade no desenvolvimento** | `groupBy` para contadores e `@@unique` para uma reação por usuário declarados no schema — sem SQL de agregação manual. |
| **No que ajudou no projeto** | `commentModel` e `reactionModel` copiaram o estilo de `experienceModel`; listagem de relatos enriquecida com contadores em lote, evitando N+1. |
| **Impacto arquitetural** | `commentModel.ts` e `reactionModel.ts` seguem o mesmo contrato de `experienceModel.ts`. |

---

### 3.2 Passport JWT + middlewares existentes

| Aspecto | Detalhe |
|---------|---------|
| **O que faz** | `authMiddleware` protege POST; `optionalAuthMiddleware` hidrata `myReaction` no GET sem bloquear visitantes. |
| **Por que a equipe reutilizou** | Infraestrutura RF01 — evita duplicar parsing de Bearer token. |
| **Facilidade no desenvolvimento** | POST comentário/reação = `authMiddleware` + `requireTurista`; GET relatos com reação pessoal = `optionalAuthMiddleware` — duas linhas na rota. |
| **No que ajudou no projeto** | Morador/admin recebem 403 sem código extra; visitante anônimo ainda lista relatos; usuário logado vê `myReaction`. |
| **Impacto arquitetural** | **Middleware chain** composável: mesma receita em denúncia e moderação posterior. |

---

### 3.3 Express — rotas aninhadas

| Aspecto | Detalhe |
|---------|---------|
| **O que faz** | Monta rotas sob `/places/:placeId/experiences/:experienceId/...`. |
| **Por que a equipe reutilizou** | Padrão já usado em relatos — URL expressa cadeia de domínio. |
| **Facilidade no desenvolvimento** | Novos endpoints (`/comments`, `/react`) adicionados no mesmo `experienceRoutes.ts` — frontend e Swagger já conheciam a estrutura. |
| **No que ajudou no projeto** | `placeAdaptor` e testes de integração não precisaram de base URL ou contrato novo para interações sociais. |
| **Impacto arquitetural** | Consistência de API — frontend e Swagger preveem estrutura uniforme. |

---

### 3.4 Vitest

| Aspecto | Detalhe |
|---------|---------|
| **O que faz** | Testa validação de texto, parser de reação e lógica de toggle. |
| **Por que a equipe reutilizou** | Runner já configurado em `vitest.config.ts`. |
| **Facilidade no desenvolvimento** | Testes de toggle de reação e validação de texto rodam em milissegundos, com mocks — sem fixture de banco. |
| **No que ajudou no projeto** | Regressão de RF12/RF13 após remover `dislike` e ajustar contadores, sem teste manual repetitivo no Swagger. |

**Arquivos de teste:** `commentService.test.ts`, `reactionService.test.ts`, `reactionTypes.test.ts`.

---

### 3.5 Código interno reutilizado (não é lib npm)

| Artefato | Origem | Facilidade / no que ajudou |
|----------|--------|----------------------------|
| `blacklist.ts` | Módulo de relatos | Mesma lista ofensiva em relatos e comentários — uma manutenção, dois services |
| `findExperienceByIdAndPlaceId` | `experienceModel` | Validação Local→Relato pronta; comentário/reação não repetem query de vínculo |
| Padrão MVC | `experience*` | Novos arquivos `comment*` / `reaction*` criados por cópia estrutural — onboarding imediato |
| `formatExperience` | `experienceView` | Contadores e `myReaction` na listagem sem endpoint extra |
| `reactionTypes.ts` | Camada anti-corrupção | API `heart`/`like` desacoplada do enum Prisma — frontend não quebra ao mudar banco |

## 4. Lógica própria — toggle de reação (RF13)

A equipe **não reutilizou** pacote de “reactions prontas” (Redis, Socket.io, lib social). A lógica em `reactionService.toggleReaction`:

1. Sem reação anterior → `INSERT`;
2. Mesma reação → `DELETE` (toggle off);
3. Reação diferente → `UPDATE` type.

Isso garante **uma reação por turista por relato** com controle fino sobre contadores.

---

## 5. Agregação eficiente na listagem

Para evitar N+1 queries, `experienceService` reutiliza Prisma `groupBy` via:

- `commentService.getCommentCountsByExperienceIds`;
- `reactionService.getReactionCountsByExperienceIds`;
- `reactionService.getUserReactionMap` (se JWT presente).

## 6. Impacto da reutilização

| Benefício | Descrição |
|-----------|-----------|
| **Zero deps novas** | `package.json` estável; menos superfície de vulnerabilidade |
| **Curva de aprendizado baixa** | Novos arquivos espelham `experience*` |
| **Consistência de moderação** | Blacklist única para relatos e comentários |
| **Auth unificada** | Mesmo JWT testável no Swagger para comment/react |

## 7. Rastreabilidade — RF12 e RF13

O texto do RF12 na especificação do projeto menciona “emoji de coração ou joia”; na implementação a equipe **separou** essas interações em dois requisitos distintos — coerente com a hierarquia REST e com responsabilidades diferentes (texto vs. toggle emoji).

| RF | Requisito | Implementação | Endpoint |
|----|-----------|---------------|----------|
| **RF12** | Comentários textuais sobre relatos de outros turistas | `ExperienceComment` + validação de texto/blacklist | `GET/POST .../comments` |
| **RF13** | Reações por emoji (coração, joia/like) | `ExperienceReaction` + toggle | `POST .../react` |

| Critério | Status | Evidência |
|----------|--------|-----------|
| RF12 — turista comenta relato alheio | Implementado | `POST .../comments` + `requireTurista` |
| RF12 — texto validado (vazio, blacklist, limite) | Implementado | `commentService.validateCommentText` + `blacklist.ts` |
| RF13 — turista reage com emoji pré-definido | Implementado | `POST .../react` — tipos `heart`, `like` |
| RF13 — toggle (mesmo emoji remove; outro troca) | Implementado | `reactionService.toggleReaction` |
| RF13 — uma reação por turista por relato | Implementado | `@@unique [experienceId, userId]` no Prisma |
| Contadores na listagem de relatos | Implementado | `GET .../experiences` — `commentsCount`, `reactions`, `myReaction` |

Documento complementar: [4.6. Comentários e Reações](/docs/requisitos/RF12-RF13-backend/4.6.ComentariosReacoes.md).
